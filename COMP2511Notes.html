Code Smells and Refactoring

Smalls: Design Aspects that violate fundamental design principles and impact software quality

Design Smells vs Code Smells

Code smells are usually not bugs; they are not technically incorrect and do not prevent the program from functioning

They indicate a weakness in design that may close down development or increase the risk of bugs or failures in the future

Regardless of the granularity, smells in general indicate the violation of software design principles, and eventually lead to code that is rigid, fragile and require “refactoring”

Code refactoring is the process of restructuring existing computer code without changing its external behaviour

Design Patterns

Creational Patterns deal with object creation mechanisms, trying to create objects in a manner suitable to the situation

  • Abstract Factory: provides an interface for creating related or dependent objects without specifying the objects’ concrete classes
  • Factory Method: allows a class to defer instantiation to subclasses
  • Builder: separates the construction of a complex object from its representation so that the same construction process can create different representations
  • Singleton: Ensures that a class only has one instance, and provides a global point of access to it

Structural Patterns are design patterns that ease the design by identifying a sim;le way to realise relationships among entities

  • Adapter: ‘adapts’ one interface for a class into one that a client expects
  • Composite: A street structure of objects where every object has the same interface (leaf and composite nodes)
  • Decorator: add additional functionality to a class at runtime where subclassing would result in an exponential rise of new classes

Behavioral Patterns as design patterns that identify common communication patterns between objects and realise these patterns.l By doing so, these patterns increase flexibility in carrying out this communication

  • Iterator: Iterators are used to access the elements of an aggregate object sequentially without exposing its underlying representation
  • Observer: Objects register to observe an event that may be raised by another object. Also known as Publish/Subscribe or Event Listener
  • State: A clean way for an object to partially change its type at runtime
  • Strategy: Algorithms can be selected at runtime, using composition
  • Template method pattern: Describes the program skeleton of a program: algorithms can be selected at runtime, using inheritance
  • Visitor: A way to separate an algorithm from an object

Abstract Classes

We can declare classes that define only part of an implementation

  • Methods can be implemented differently in different subclasses of the abstract class

Rules:

  1. An abstract class is declared abstract
  2. If class includes abstract methods, class must be declared abstract
  3. Abstract class cannot be instantiated
  4. Subclass of an abstract class can be instantiated if it overrides each of the abstract methods and provides an implementation for all of them
  5. If subclass of abstract class doesn’t implement all abstract methods, the subclass is itself abstract

Single Inheritance versus Multiple Inheritance

  • A new class can extend exactly one superclass - single inheritance
  • In Java, interface in the class hierarchy can be used to add multiple inheritance

Interfaces in Java

  • Like abstract classes, but few important differences
  • All methods defined are implicitly abstract (no need for abstract keyword, but it can be used for clarity)
  • Variables declared in an interface must be static and final (must be constants)
  • A class can implement an interface
  • A class must declare the interface in an implements clause
  • Then, provide an implementation for all abstract methods in interface
  • A class can implement more than one interface

Interfaces can have sub-interfaces

Sub-interface inherits all the abstract methods and constants of its super-interface and may define new abstract methods and constants

  • interfaces can extend more than one interface at a time

Design Smells

  • A symptom of poor design

Rigidity

  • Tendency of the software being too difficult to change even in simple ways
  • A single change causes a cascade of changes to other dependent modules        

Fragility

  • The tendency of the software to break in many places when a single change is made
  • Rigidity and fragility complement each other

Immobility

  • Design is hard to reuse
  • Design has parts that could be useful to other systems, but the effort needed and risk in disentangling the system is too high

Viscosity

  • Changes are easier to implement through “hacks” over design preserving methods
  • Environment viscosity - development environment is slow and in-efficient

Opacity

  • Tendency of a module to be difficult to understand
  • Code must be written in a clear and expressive manner

Needless complexity

  • Contains constructs that are not currently useful
  • Developers Ahead of requirements

Needless repetition

  • Design contains repeated structures that could potentially be unified under a single abstraction
  • Bugs found in repeated units have to be fixed in every repetition

Characteristics of good design

Loose Coupling and High Cohesion

Software entities are:

  • Extensible
  • Reusable
  • Maintainable
  • Understandable
  • Testable

Coupling

  • Interdependence between components or classes
  • High coupling occurs when one component depends on the internal workings of another component B, and is affected by internal changes to component B
  • Leads to complex system with difficulties in maintenance and extension

Loose coupling - allows components to be used and modified independently of each other

  • But, Zero-coupled classes are not usable

Cohesion

  • The degree to which all elements of a component, class or module work together as a functional unit

Highly cohesive modules:

  • Easier to maintain
  • Less frequently changed
  • Higher probability of reusability

Don’t put all  responsibility into a single class to avoid low cohesion

Refactoring

The process of restructuring (Changing the internal structure of software) software to make it easier to understand and cheaper to modify without changing its external, observable behaviour

  1. Improve design to software
  2. Improve code readability
  3. Find Bugs
  4. Helps you program faster - improved maintenance and extensibility
  5. To conform to design principles
  6. Avoid design smells
  7. Reduced complexity

  • Helps identify hidden or dormant bugs or vulnerabilities by simplifying code logic
  • Different to adding features and debugging

When to refactor

Refactor when:

  • You add a function
  • When you need to fix a bug

Motivation

  • Automatic unit tests should be set up before refactoring to ensure routines still behave as expected

#3 Observer Pattern

Type

Behavioural

Purpose

Used to implement distributed event handling systems, in event driven programming

In the observer pattern:

  • An object called the subject (observable/publisher) maintains a list of its dependents, called observers (or subscribers)
  • Notifies the observers automatically of any state changes in the subject, usually by calling one of their methods

One-to-many dependency between objects so that when on object (subject) changes state, all its dependents (observers) are notified and updated automatically

Aim:

  • Define a 1-many dependency between objects without making them tightly coupled
  • Automatically notify/update an open-ended number of observers (dependent objects) when the subject changes state
  • Be able to dynamically add and remove observers

UML

Advantages

Avoids Tight coupling between Subjects and its observers

  • Allows subject and its observers to be at different levels of abstractions in a system
  • Loosely coupled objects are easier to maintain and reuse
  • Allows dynamic registration and deregistration

Be careful

  • Change in the subject may result in a chain of updates to its observers and their dependent objects
  • complex update behaviour
  • Need to properly manage dependencies

Implementation

  • Define Subject and Observer interfaces, such that when a subject changes atte, all registered observers are notified and updated automatically

Subject: Maintains a list of observers and notifies them of state changes by calling their update() operation

Observers: Register and unregister themselves on a subject (to get notified of state changes) and to update their state when they are notified

  • This makes subject and observers loosely coupled
  • Observer can be added and removed independently at run-time
  • Publish-subscribe

Passing Data:

Push or Pull

Subject needs to pass(change) data while notifying a change to an observer.

Push Data:

  • Subject passes the changed data to its observers
  • All observers must implement the above update method

Pull Data:

  • Subject passes a reference to itself to its observers, and observers need to get (pull) the required data from the subject
  • Subject needs to provide the required access methods for its observers

#4 Composite Pattern

Type: Structural Pattern

Aim:

A composite is an object design as a composition of one-or-more similar objects

To be able to manipulate a single instance of the object, just as we would manipulate a group of them

  • Resizing a group of shapes should be the same as resizing a single shape
  • No discrimination between a single (leaf) vs a composite (group) object

Anatomy

Leaf: Object performs operations on the object

Composite: Object performs operations on its children, and if required collects return values and derives the required answers

Uniformity vs Type Safe

Type Safe (LEFT)

  • Only define child-related operations in the composite class
  • Enforces type constraints - client cannot perform child-related operations on a leaf object
  • Client needs to treat leaf and composite objects differently
  • Useful for static structures where a client doesn’t need to perform child-related operations on “unknown” objects of type component

Uniformity (RIGHT)

  • Include all child-related operations in the component interface:
  • Leaf class implements methods with “do nothing” or “throw an exception”
  • Client can treat leaf and composite uniformly
  • Loose type safety because leaf and composite types are not cleanly separated
  • Useful for dynamic structures where children types change dynamically (from Leaf to Composite) and client needs to perform child-related operations regularly

UML

Example

Summary

  • Composite pattern provides a structure to hold both individual objects and composites
  • Allows clients to treat composites and individual objects uniformly
  • Component is: any object in a composite structure
  • May be other composites or leaf notes
  • There are many design tradeoffs in implementing composite
  • Need to balance transparency/uniformity and type safety

#5 Iterator Pattern

Type

Behavioural

Purpose

“Provide a way to access the elements of an aggregate (many parts) object sequentially without exposing its underlying representation”

Make new object of type iterator, put all data into that object - whatever object you make, you can make an iterator

  • Provides hasNext() and next()

Addresses:

Exposing representation details of an aggregate breaks its encapsulation

How can the elements of an aggregate be accessed and traversed without exposing underlying representation

  • An iterator is usually implemented as an inner class of aggregate
  • Enables access to all fields of class
  • New access and traversal operations can be added by defining new iterators:
  • E.g. back to front: previous(), hasPrevious()
  • Aggregate provides an interface for creating an iterator createIterator()
  • Multiple traversals is possible (simultaneous traversals)

#8 Template Pattern

Goal

Define the skeleton of an algorithm in an operation, deferring some steps to subclasses.

  • Implements invariant parts
  • Calls primitive operations that could be implemented by sub classes OR has default implementations in an abstract super class
  • Redefine certain parts of behaviour without changing other parts of the structure of the behaviour

Purpose

Lets subclasses redefine certain steps of an algorithm without changing the algorithm’s structure

 Motivation

Subclasses do not control the behaviour of a parent class, a parent class class the operations of a subclass and not the other way around

Inversion of control:

  • When using a library (reusable classes) we all the code we want to reuse
  • When using a framework (Template pattern) we write subclasses and implement invariant code to the framework class
  • Implement common (invariant) parts of a behaviour once, and leave it up to subclasses to implement the behaviour that can vary
  • Invariant behaviour is in one class (localised)

Terms:

To reuse abstract class effectively, we must understand what will be overridden.

 

Primitive operations: OPerations that have default implementations or must be implemented by subclasses

Final operations: Concrete operations that cannot be overridden by subclasses

Hook operations: Concrete operations that do nothing by default and can be redefined by subclasses if necessary. This gives subclasses the ability to “hook into” the algorithm at various points, if they wish; a subclass is also free to ignore the hook

#13 Visitor Pattern

Type

Behavioural

Goal

Separating an algorithm from an object on which it operates

  • Add new operations to object without modifying its structure
  • Follow Open/Closed principle

Purpose

Adds new operations/behaviours to the existing objects without modifying them

Visitor

  • Implements all the appropriate specialisations of the virtual operation/method
  • Takes instance reference as input and implements the goal (additional behaviour)
  • Visitor pattern can be added to public APIs, allowing its clients to perform operations on a class without having to modify the source

Problem

A geographical information structured as one colossal graph

  • each node of the graph represents a city, an industry, sightseeing area etc
  • Each node type is represented by its own class, each specific node is an object
  • Task: Add behaviour to each node → to export graph into XML format

Solution

Visitor pattern suggests that you place the new behaviour into a separate class called visitor instead of trying to integrate it into existing classes

  • Original object that had to perform the behaviour is now passed into the visitor’s methods as an argument
  • All necessary information must be contained within object
  • Visitor class needs to define a set of methods, one for each type of Node
  • Visitor pattern uses “Double Dispatch” to execute a suitable method on a given object
  • Object accepts a visitor and tells it what visiting method should be executed

Design By Contract

Defensive Programming

Tries to address unforeseen circumstances, in order to ensure the continuing functionality of the software element.

For example, it makes the software behave in a predictable manner despite unexpected actions.

  • Used where high availability, safety or security is needed
  • Results in redundant checks (both client and supplier may perform checks) and more complex software for maintenance
  • Difficult to locate errors , considering there is no clear demarcation of responsibilities
  • Mya safeguard against errors that will never be encountered, incurring run-time and maintenance costs

Design By Contract

At design time, responsibilities are clearly assigns to different software elements, clearly documented and enforced during the development using unit testing and/or language support

  • Clear demarcation of responsibilities help prevent redundant checks
  • Results in simpler code and easier to maintain
  • Crashes if the required conditions are not satisfied
  • May not be suitable for high availability applications         

A contract (precondition, post-condition) and invariant should be:

  • Declarative and must not include implementation details
  • Precise, formal and verifiable

Every software elements should define a specification (or a contract) that governs its interaction with the rest of the software components

A contract should address:

1

Pre-condition

What does the contract expect?

If the precondition is true, it can avoid handling cases outside of the precondition

E.g. Expected argument value (mark >= 0) and (marks <= 100)

2

Post-condition

What does the contract guarantee?

Return value(s) is guaranteed, provided the precondition is true

E.g. Correct return value representing a grade

3

Invariant

What does the contract maintain?

Some values must satisfy constraints before and after the execution


E.g. A Value of mark remains from 0-100

Benefits

  • No error checking required for conditions that do not satisfy preconditions
  • Prevents redundant validation tasks
  • Given preconditions are satisfied, clients can expect the specified post-conditions
  • Responsibilities are clearly assigned, this helps in locating errors and resulting in easier code maintenance
  • Helps in cleaner and faster development

Implementation Issues

  • Some programming languages offer native support for DbC
  • Java does not have native support for DbC
  • In the absence of native language support, unit testing is used to test the contracts
  • Often, preconditions, postconditions and invariants are included in the documentation

Example

Pre-conditions

  • A precondition is a condition (or predicate) that must always be true, prior to the execution of some section of code
  • If a precondition is violated, the effect of the section of code becomes undefined and may not carry out its intended work
  • Security problems can arise due to incorrect pre-conditions
  • Foten, preconditions are included in the documentation of the affected section code
  • Preconditions are sometimes testing using guards or assertions within the code itself, and some languages have specific syntax constructions for testing

Example:

  • In a subclass: Pre-condition can be weaker, but can’t be stronger

Post-conditions

A condition that must always be true after the execution of some section of code

  • Post-condition is a declaration of the properties which are guaranteed upon completion of the routine’s execution
  • Sometimes: Post conditions are tested using guards or assertions within the code itself and some languages have specific syntax constructions for testing

  • Post-condition may be strengthened (more restricted)

Class Invariant

Constraints the state (values of certain variables) stored in the object)

  • Class invariants are established during construction and constantly maintained between class to public methods.
  • Methods of the class must ensure class invariants are satisfied/preserved
  • Within a method: code may break invariants as long as the invariants are restored before the method ends
  • Class invariants help programmers to rely on a valid state, avoiding the risk of inaccurate/invalid data
  • Helps locating errors during testing
